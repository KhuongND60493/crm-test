You are generating a **Cursor-compatible task file** from a Claude Code SPEC or PLAN.

## Purpose

In the dual-AI workflow:
- **Claude Code** is the **Architect** — it designs via `/ask` → `/spec` → `/plan`
- **Cursor AI** is the **Implementer** — it executes the plan via `/implement`

This command is the **bridge**: it takes Claude Code's approved SPEC or PLAN
and produces a `tasks_*.md` file in the **exact format** that Cursor's Rule 04 (`/implement`) expects.

```
Claude Code                          Cursor AI
──────────────────────────────────────────────────
/ask → /spec → /plan                    /implement
    [APPROVED CODE]                         ↑
         ↓                                  │
   /generate-cursor-task ─────────────────→ tasks_[name].md
```

---

## How To Use

```
/generate-cursor-task [spec-or-plan-file]
```

Examples:
```
/generate-cursor-task .ai/plans/PLAN-order-cancellation.md
/generate-cursor-task .ai/specs/SPEC-user-login-otp.md
/generate-cursor-task          ← auto-find the most recently approved PLAN
```

---

## Step 1 — Resolve Source File

**If a file path is given:** Read that file.

**If no path given:** Look for the most recently modified file in `.ai/plans/`
(plans are preferred — they have more implementation detail than specs alone).
If `.ai/plans/` is empty, look in `.ai/specs/`.

If nothing found:
> "⚠️ No SPEC or PLAN found. Run `/spec` and `/plan` first, then come back."

---

## Step 2 — Read Context

Also read:
- `docs/project_overview.md` — for correct file paths, naming conventions, project structure
- The corresponding SPEC if a PLAN was provided (for business rules and acceptance criteria)

---

## Step 3 — Validate Approval Status

Check the PLAN/SPEC status:
- If PLAN has `**Status:** Approved` or the user is running this command (implying they approved it) → proceed
- If PLAN still says `Draft` → warn:

```
⚠️ This PLAN is still in Draft status.
   In the ROMIO workflow, run /plan, review it, then reply APPROVED CODE before generating cursor tasks.
   
   Proceed anyway? (yes / no)
```

## Step 4 — Generate Cursor Task File

Create file at: `docs/tasks/tasks_[feature-name].md`

Use the **exact format** that Cursor's `04_implementation.md` Rule expects:

```markdown
# Task Breakdown: [Feature Name]

> TDD Reference: [path to SPEC file]
> PLAN Reference: [path to PLAN file]
> Generated by: Claude Code (/generate-cursor-task)
> Created: [DATE]
> Status: In Progress
>
> Progress: 0 / [N] tasks completed

---
```

Then generate each section from the PLAN content:

### Mapping Rules: PLAN → Cursor Tasks

| PLAN Section | Cursor Task Section |
|---|---|
| §7 Data Model Changes | `## 🗄️ Database` — Task DB-01, DB-02... |
| Files to change: Domain entities | `## 🏛️ Domain Layer` — Task DM-01, DM-02... |
| Files to change: Application handlers | `## 📦 Application Layer` — Task APP-01, APP-02... |
| Files to change: Infrastructure | `## 🔧 Infrastructure Layer` — Task INF-01... |
| §8 API Changes + Presentation files | `## 🌐 API / Presentation Layer` — Task API-01... |
| §10 Test Plan | `## 🧪 Testing` — Task TEST-01... |
| From SPEC §10 Acceptance Criteria | `## 📝 Documentation` — Task DOC-01... |

### Task Format (exact Cursor-compatible format)

Each task must have:
```markdown
- [ ] Task [PREFIX]-[N]: [Action verb] `[ClassName or FileName]`
  - File: `[exact/file/path/from/project]`
  - [Relevant detail: properties, methods, logic steps, SQL, etc.]
  - [Sub-step if needed]:
    - [ ] [sub-step 1]
    - [ ] [sub-step 2]
```

Rules:
- **File paths must be exact** — copy from PLAN section "Files Expected To Change"
- **Action verbs**: Create / Implement / Add / Modify / Register / Migrate
- **Task IDs**: DB-01, DM-01, APP-01, INF-01, API-01, TEST-01, DOC-01
- **Order**: DB → DM → APP → INF → API → TEST → DOC (dependencies first)
- **Granularity**: each task = ~2-4 hours of work. Split if larger.
- **Omit empty sections**: if no DB changes, don't include the DB section

### For Tasks with Logic Steps

For Application handlers and complex tasks, expand the logic from the PLAN:

```markdown
- [ ] Task APP-03: Implement `[HandlerName]`
  - File: `[exact path]`
  - Inject: [IRepository, IUnitOfWork, etc. from PLAN]
  - Logic:
    - [ ] [Step 1 from PLAN §6]
    - [ ] [Step 2]
    - [ ] [Step N]
    - [ ] Return `Result<[Type]>`
```

### For DB Tasks

```markdown
- [ ] Task DB-01: Create EF Core migration `[MigrationName]`
  - Changes: [from PLAN §7 — tables/columns added/modified]
  - Command: `dotnet ef migrations add [MigrationName] --project [InfraProject]`
  - ⚠️ Review generated migration script before applying
```
